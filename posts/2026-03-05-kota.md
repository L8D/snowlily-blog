# Kota

Kota is the first of many-to-come tools I'm crafting for my own use within my
homelab.

The purpose of Kota is to provide the MVP of a task manager like Linear.
Coincidentally, the team at [HumanLayer](https://humanlayer.dev/) has something
similar in-house which is used in combination with Linear.

Kota is, at its core, a CLI tool. It provides an interface for:

- Creating "tickets" with titles, descriptions and auto-generated sequential IDs
- Adding "documents" to tickets
- Using the filesystem as the source of truth for tickets
- Using git behind-the-scenes to maintain a revision history

I ended up with this exact set of functionality by distilling down the most-essential set of features that I was using Linear for. All of this functionality is already present in Linear.

## Why build Kota instead of using something off the shelf (Linear, Huly, Plane)?

Part of a theme for my home lab is a sense of self-dependence. I am wanting to build out a monorepo that contains everything needed to reproducibly set up a fresh home lab server from scratch, and everything I would need for normal day-to-day experimentation would be self-hosted and part of the monorepo (there are necessary exceptions to this, since I don't have the hardware needed to run local LLMs, yet, so I've been using cloud-hosted models).

At first, I went looking for an open-source self-hostable task manager. I spent a little bit of time considering [Plane](https://plane.so/) and [Huly](https://huly.io/) but after briefly considering "what functionality am I _really_ depending on?" I realized that I could build out the most essential functionality within an hour or less myself.

I started looking into the self-hosting requirements for the aforementioned task managers, and saw all sorts of things like needing to set up and configure postgres or redis or fancy docker things with network management and auth and sign in— etc. It felt like I would be spending time on all of this, and spending time on trying to recreate the CLI-based functionality from [Linearis](https://github.com/czottmann/linearis). In a moment of clarity, I did realize that it was in my best interest to just build something like Kota quickly and move on.

Also worth mentioning is that there are dozens, if not hundreds of filesystem-based task managers out there, especially since the rise of "spec-driven development" with agentic coding tools. Some of the more popular examples include:

- https://github.com/eyaltoledano/claude-task-master
- https://github.com/unnecessary-special-projects/ghist
- https://github.com/MrLesk/Backlog.md
- https://github.com/GothenburgBitFactory/taskwarrior
- https://github.com/obsidian-tasks-group/obsidian-tasks
- https://github.com/git-bug/git-bug
- https://github.com/BaldissaraMatheus/Tasks.md

Kota is yet-another one of these tools. I have tried `claude-task-master` once before because it was specifically referred to me and appeared to be good-enough and better-than-nothing. But generally, I would hesitate to reach out for one of these tools because you will find yourself building a lot of tooling around them and it can just become such a pain to switch tools or gain control over how the tool actually works.

For example, I quickly ran into huge undesirable shortfalls with `claude-task-master` when I tried to use it. There were plenty of things that were less-than-ideal and then plenty of bells and whistles that I wouldn't find myself using any time soon. It was just a tool that was very opinionated (more opinionated than Linear) and was trying to do too many different things. Hypothetically, I wouldn't want to prescribe something like that to a team of engineers and start to see them trying to utilize many of the less-than-optimal bell and whistles instead of reaching for more purpose-specific tools that do the same jobs but better.

By building my own tool, I can keep all the unnecessary bells and whistles out of the way and focus on the core functionality that I need. If I ever run into a shortfall, I can fix it, and the overall total implementation of Kota is small enough that I can understand how to fix it too.

## Why use the filesystem as the source of truth?

Many times in the past, especially when using `linearis` with an LLM agent, I've found that LLMs are great at working with markdown in files. Actually, that is what they tend to do by default when managing documents and stuff. So there would occasionally be an "awkwardness" where the agent would try to use a big Bash snippet with a heredoc in order to move content around and edit it in bulk with the corresponding CLI tool.

Whenever the awkwardness arose, the thought would occur to me, "wouldn't it be great if Linear had an interface where the tickets are synced to the filesystem, so that a model could interface with it more naturally?"— meaning that a process like `rsync` or a filewatcher would automatically relay changes to Linear and vice versa. Now you can point the LLM at the right files, and it would manage your tasks without even knowing that they were being persisted to Linear under the hood.

In part of trying to approach a design that was simple enough to build quickly, I knew that starting with the file system just "made sense." Using a database like sqlite felt like overkill since all we're doing is creating sequential identifiers, then working with a plaintext title and a markdown description. Technically, I haven't immediately jumped to the idea that I've previously envisioned, which was to point the LLM agent at the filesystem— because part of my goal was to stick to a linearis-like CLI interface so that I would be builting tooling around that which could later be repurposed to work with linearis one day.

I could talk on and on here. There are lots of little reasons. Having mostly-plaintext content makes it feel simple and straightforward to automate the git commands necessary to create a revision history. Linear has a revision history feature that I have found useful, and so I wanted to retain that functionality.

The idea of using a filesystem is also appealing in simplicity, especially if ever need to keep data synced between my workstation and my home lab server, I see `rsync` as a sturdy, simple and reliable way to do that. So it feels like I'm leaving the door open for the possible future situation where I'm using my home lab server to run through agentic workflows that alos need access to the task manager.

## How was Kota built?

Kota is remarkably simple. It is just a bash script. At the time of writing this, it is only ~1000 lines of code. 

As mentioned above, when tickets are created or updated, Kota just writes to folder in the filesystem and immediately auto-writes a git commit. The "kota home" folder is auto-initialized as a git repo.

## What is it like to use Kota?

Here is the help output, at the time of writing this:

```
kota - A CLI tool for managing tickets and documents, backed by a local git repository.

COMMANDS:
  kota usage, kota --help     Show this help message
  kota init                   Initialize the kota data directory and git repo

TICKETS:
  kota tickets create         Create a new ticket
  kota tickets read <id>      Read a ticket as JSON
  kota tickets update <id>    Update an existing ticket
  kota tickets delete <id>    Delete a ticket
  kota tickets list           List all tickets

DOCUMENTS:
  kota documents create       Create a new document attached to a ticket
  kota documents read <id>    Read a document as JSON
  kota documents update <id>  Update an existing document
  kota documents delete <id>  Delete a document
  kota documents list         List documents (optionally scoped to a ticket)

ENVIRONMENT VARIABLES:
  KOTA_HOME                   Path to the kota data directory (git repo). Required.
  KOTA_DEFAULT_PREFIX         Default ticket prefix (e.g. DEV) for ticket creation.

EXAMPLES:
  kota init
  kota tickets create --title "Fix login bug" --description "Users cannot log in"
  kota tickets create --title "New feature" --prefix FEATURE
  kota tickets read DEV-123
  kota tickets update DEV-123 --title "Updated title"
  kota tickets list
  kota documents create --ticket DEV-123 --title "Design notes" --content "Notes here"
  kota documents read DEV-123/cfe75b5c-86e9-46d7-8a38-04a9884b5b97
  kota documents list --ticket DEV-123
```

I use [direnv](https://github.com/direnv/direnv) for my projects (worth mentioning, [quickenv](https://codeberg.org/untitaker/quickenv) is a drop-in replacement that I will probably find myself switching to in the future), with a `.env.local` file that gets loaded into my dev environment. As a result, I can configure the `KOTA_HOME` and `KOTA_DEFAULT_PREFIX` values on per-project or per-folder basis. 

This is how I would add a claude skill for kota (assuming you've already configured `KOTA_DEFAULT_PREFIX`) so that claude would just magically know what to do if you put a ticket ID in a prompt somewhere.

```bash
mkdir -p .claude/skills/kota
tee .claude/skills/kota/SKILL.md <<EOF
name: kota
description: Use when reading, creating or updating kota tickets/documents. Always use this skill when you see $KOTA_DEFAULT_PREFIX-* ticket IDs or need to interact with Kota in any way.
---

$(kota usage)
EOF
```
